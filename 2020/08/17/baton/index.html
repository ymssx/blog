前端请求合并方案的思考

2020年8月18日 08:00 # 浙江-杭州
计算机网络 javascript

# 前端请求性能优化

我在接触一些大型前端项目的时候，发现页面初始化加载太慢是一个常见的问题。我简单的看了一下Network面板，发现主要存在以下两个问题：

  1. 前端请求的并发限制。浏览器给http连接最高限制是6个，导致许多请求需要排队等待。
  2. 请求之间的依赖关系。比如b请求需要在a请求的结果基础上做处理，然后再发出b请求，这个时候b请求就需要等待a的一次网络来回io才能被发起，这个过程就浪费了一次等待周期。特别是当请求依赖树过于庞大的时候，整个页面就会存在大量的等待时间以及不必要的网络io。

关于这一点，其实我们可以从优化接口的角度去改进。比如在优化接口之后，后端一次性返回a、b的全部数据，从而节省了一次网络io。但是实际情况下前端的请求依赖可能远比a、b之间的关系复杂，将优化工作丢给后端会增加大量的前后端沟通成本。所以有没有一种一劳永逸的方法，让前后端开发者更专注于自己的业务逻辑呢？

# 请求合并小工具

一个自然的想法是，在前端对fetch进行封装，通过在一次主任务中将所有请求搜集起来，包括它们之间的依赖逻辑、计算逻辑，请求并不及时发送，而在一定的延时后（setTimeout或者Promise等，我选择setTimeout，后续会详细讲）合并发送一个请求树；

而在后端，服务器在收到请求树之后，将其解析，并逐步处理后统一返回。值得注意的是，请求之间不仅是逻辑上的依赖关系，更多的是计算的依赖关系，比如b请求需要对a请求的结果作运算之后才能发起b请求，而我们在服务端统一计算的时候自然也需要进行这个计算，所以我们需要在后端运行前端js代码。

总结一下，本思路的几个点如下：

# 前端

  1. 前端封装了一个自己的请求api，这个api并不会及时发起请求，而是收集在一个请求树里面。在异步队列中我们统一将所有的请求发出，包括他们的依赖关系。
  2. 一个请求b往往会依赖另一个请求a的结果，在使用请求api的时候，需要描述自己对a.result的处理逻辑，然后返回一个新的请求url。这个时候，该api会自动记录这个计算逻辑function。
  3. 请求a、b、c等相互依赖，最终会形成一个依赖树，每个节点描述了如何利用自己父节点的result进行运算，也就是说，每个节点记录了一个function: pre => url。
  4. 既然是一颗树，那么就会有叶子节点，叶子节点也就是最终我们需要的数据。比如a => b => c，那么c就是一个叶子节点。
  5. 在setTimeout之后，我们会将这个树转换为json文本发送给后端。使用setTimout是为了能够方便的收集用户交互时产生的请求依赖。

# 后端

  1. 后端收到的是一个请求树，会将其解析，以a => b => c为例。
  2. 首先对a的url进行路由映射，计算a的结果并返回a.result。这是我们开始计算b，由于b需要对a的结果进行处理，我们去除b节点的计算function，将a.result作为参数输入函数，返回一个新的b.url，然后我们将b.url进行请求获取b.result，然后作为输入计算c。
  3. 当a、b、c计算结束后，我们会得到一个响应树，将这个响应树进行统一返回。在前端对节点a、b、c进行resolve，返回其需要的结果。

# 前端远程攻击的危险

因为服务端需要执行前端代码，而前端代码具有不可知性，所以服务器存在远程攻击的风险。

一个解决思路是使用进程隔离，另一个思路是使用沙箱隔离，我选择了后者，因为实现起来比较容易。

# 请求、响应内容进行压缩

对于请求内容，其存在大量的function代码，所以我们可以对js代码进行压缩，比如缩短变量名、删减空格、treeshaking等。

对于响应内容，我们可以在前端规定好我们需要哪些字段，最终返回的时候按需返回即可。

# 使用示例

前端

import start from './baton.js';

const root = start('http://localhost:3000/baton/');

const a = root.next({ path: '/aaa' });

const x = 1;
const y = { m: 2 };

const b = a.next((res) => {
  const { i, j } = res;
  return {
    path: '/bbb',
    params: {
      i: i + j,
      j: x,
    },
  }
}, { x, y });

const c = await b.get(res => {
  return {
    ...res,
    ok: 'ok',
  };
});

console.log(c);

后端

const Koa = require('koa');
const { BatonServer } = require('./batonServer');
const bodyParser = require('koa-bodyparser');

const KoaApp = new Koa();
KoaApp.use(bodyParser());
const app = new BatonServer(KoaApp, '/baton');

app.get('/aaa', async ctx => {
  return {
    i: 1,
    j: 2,
  }
});

app.get('/bbb', async ctx => {
  const { params } = ctx.request;
  const { i, j } = params;
  return {
    i: 3 + i,
    j: 4 * j,
  }
});

// 在端口3000监听:
app.listen(3000);
